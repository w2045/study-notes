# Study 项目工作流

## 启动协议（每次会话开始）

1. 读 `CLAUDE.md` 了解项目约定
2. 读 `PROGRESS.md` 获取当前进度与中断点
3. 读当前学科的 `OUTLINE.md`（如 `数学基础/微积分/OUTLINE.md`）
4. 向用户报告：当前在哪、下一步是什么、上次改了什么
5. 等用户确认后开始

## 结束协议（每轮工作完成后）

1. 内容落盘
2. 更新 `PROGRESS.md`：标记已完成章节、更新「下次从 X 开始」、在「本次会话记录」追加条目
3. `git add` 具体文件（不用 `-A`）+ commit + push origin main

## 每章结构

### 数学类（线性代数 / 微积分 / 概率论 / 凸优化 / 信息论）

含编程的章节：
```
XX-章节名/
├── notes.md
├── 理论题/
│   ├── homework.md
│   └── solutions.md
└── 编程题/
    ├── homework.md
    ├── homework.py
    ├── grader.py
    └── solutions.md
```

纯理论章节：去掉 `编程题/`，只留 `理论题/`。

### 编程类（Python基础 / 数据结构与算法）

```
XX-章节名/
├── notes.md
└── 作业/

    ├── grader.py        # importlib 动态导入，math.isclose 浮点比较
    └── solutions.md     # <details> 折叠答案
```

### 工具类（基础工具）

```
XX-章节名/
```

## notes.md 编写规范

### 核心原则

以「一个仅具备单变量微积分 + 基础编程的本科生能否从当前内容无痛学会」为标准。

### 每个概念的 5 要素链

1. **动机**（1-3 句）：这个概念回答什么问题？为什么需要它？
2. **形式定义**：用 `<blockquote>` 定义框
3. **手算/代码示例**：从零推导。算法不能只用平凡输入演示
4. **性质/定理**：关键推论
5. **应用连接**：在哪里用到？给出具体场景

### 小节衔接

每个新小节出现前，必须有 1-3 句过渡解释：
- 前一小节讲了什么
- 为什么接下来要讲这个
- 它和前文有什么关系

概念不能「凭空出现」——每个知识点出现时，前文已做好铺垫。

### 跨章引用

- 数学类：`ChXX §Y.Y` 格式，一章是主定义位置，其余章引用
- 编程类：正文中自然引用（「这本质是第三章 §2 的闭包机制」）
- 每章顶部 `← 前置` 和底部 `→ 延伸` 导航链接

### 易错点

随概念讲解自然带出，不单独列表。在读者可能踩坑的地方立刻说明。

### 每章结尾

- 例题 2-4 个，`<details>` 折叠答案
- 核心概念速查表

## homework.md / grader.py / solutions.md 规范

- homework.md: ≥10 题，标注 ⭐/⭐⭐/⭐⭐⭐，题型混合（补全、实现、找 bug、doctest、综合）
- homework.py: 函数签名 + docstring（含 doctest），函数体 `pass` 占位，顶部注 `python3 grader.py`
- grader.py: `import importlib` 动态导入，多组测试含边界，`math.isclose` 浮点比较，✅/❌ 输出 + 总分
- solutions.md: `<details><summary>` 折叠，每题附「要点」解释关键思路

- [ ] 跨章引用正确？
