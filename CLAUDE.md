### notes.md 规范
1. `← 前置: [章节]` 和 `→ 延伸: [章节]` 在顶部和底部
2. 直觉引入 → 形式定义 → 推导/证明 → 代码 → 例题 → 常见误区
3. 数学用 `$$` / `$` 嵌入。代码配上逐行解释
4. 末尾附「本章核心概念速查」表

### 教学内容质量标准（所有学科通用）

**核心原则**：以「一个仅具备单变量微积分 + 基础编程的本科生能否从当前内容无痛学会」为标准。

#### 每个概念的必备要素
1. **动机**（1-3 句）：这个概念回答什么问题？为什么需要它？没有它会怎样？
2. **形式定义**：用 `<blockquote>` 定义框，编号统一
3. **手算示例**：至少一个从零推导的计算示例。算法类概念必须演示非平凡输入（不要用「已经是答案」的输入）
4. **性质/定理**：列出关键推论
5. **应用连接**：这个概念在哪里用到？给出具体场景（ML/CV/物理等）

#### 必须避免的问题
- ❌ 只有定义没有计算：概念定义后必须立刻跟手算示例
- ❌ 算法只演示平凡输入：Jordan 型不能全用「已经是 Jordan 型」的矩阵；消元法不能全用「已经是上三角」的矩阵
- ❌ 缺少跨章连接：概念出现在多章时，必须标注「主定义位置」和「引用位置」
- ❌ 只有公式不给推导：关键公式必须给逐步推导，每一步标注用到的规则
- ❌ 缺失核心概念：检查是否有「只定义但从未展开」的概念（如只在一行表格里出现伪逆、只提名字没给计算的重数）

#### 每章结构（灵活，非模板）

结构服务于内容，不需要每章机械套用。通常包含：

- **开头**：直觉引入 + 核心洞察（为什么需要这个？）
- **中间**：概念讲解循环——动机 → 定义 → 手算/代码示例 → 深入细节
  - 易错点和边界情况**随概念讲解自然带出**，不单独列表
  - 跨章引用写在正文中（「这本质是 Ch03 §2 的闭包」）
- **结尾**：例题（2-4 个，`<details>` 折叠答案） + 核心概念速查表

#### 数字与引用规范
- 定义编号全局统一在每章内
- 跨章引用格式：`ChXX §Y.Y`
- 同一概念出现在多章时：一章是「主定义位置」，其余章引用它

#### 审核检查清单
修改或新增章节后，逐项验证：
- [ ] 每个新概念有手算示例？
- [ ] 算法演示使用了非平凡输入？
- [ ] 所有引用的概念在前面章节已定义？
- [ ] 跨章引用链接正确？
- [ ] LaTeX 公式渲染正确？
- [ ] 本科生能否从本节学会？（最重要的一条）

### homework.md 规范
- ≥10 题，标注难度 ⭐/⭐⭐/⭐⭐⭐
- 题型混合：代码补全、实现函数、找 bug、doctest、综合题
- 每题给出公开测试用例

### homework.py 规范
- 函数签名 + docstring（含 doctest）
- 函数体用 `pass` 占位
- 顶部注明运行方式：`python3 grader.py`

### grader.py 规范
- `import importlib` 动态导入 `homework.py`
- 每道题多组测试用例（含边界）定义在 `TEST_CASES`
- 浮点数比较用 `math.isclose`
- 输出每道题的 ✅/❌ + 错误详情 + 总分

### solutions.md 规范
- 所有答案用 `<details><summary>` 折叠
- 每题附「要点」解释关键思路和常见错误
